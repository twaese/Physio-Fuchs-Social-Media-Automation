# WF-02 v18.3 — Foto-Branch (variable Praxis-Fotos pro Content)

**Stand:** 2026-06-08
**Status:** Spec — bereit zur Implementierung in n8n
**Vorgänger:** v18.2 (statisches Hintergrund-Foto hardcoded in HTML-Template)

---

## Was sich ändert

In v18.2 wurde im HTML-Template `assets/judith-behandlung-01.png` als Hintergrund hardcoded gerendert.
Mit v18.3 setzt WF-02 **pro Content-Item ein anderes Foto** ein, das Judith vorab im SharePoint hinterlegt hat.

```
v18.2 Pipeline (statisch):
   HTML-Template (Foto hardcoded) → Gotenberg → PNG

v18.3 Pipeline (variabel):
   SP-Item lesen
      ↓
   Foto in SP suchen (per ItemID + Jahr)
      ↓
   Foto nach GitHub pushen (für public URL)
      ↓
   HTML-Template mit {{BACKGROUND_PHOTO}} = GitHub-URL füllen
      ↓
   Gotenberg → PNG
```

---

## SP-Konvention

**Pfad:** `Content_Socialmedia/Fotos/{year}/`
(„`Physio Fuchs Administration - Socialmedia`" → „Content_Socialmedia" → „Fotos" → Jahres-Unterordner)

**Filename:** `PF_{year}_{itemId-3stellig}_{thema-lowercase}.{jpg,jpeg,png}`

Beispiele:
- `PF_2026_010_schuhe_fuesse.jpg`
- `PF_2026_011_atmung.jpg`
- `PF_2026_017_nacken.jpg`

→ ItemID immer 3-stellig mit führenden Nullen (`010`, nicht `10`).
→ Thema klein geschrieben, Umlaute durch Doppelvokal (`fuesse`), keine Leerzeichen.
→ Varianten erlaubt durch Suffix: `PF_2026_010_schuhe_v2.jpg` (WF-02 nimmt alphabetisch erstes).

Details: `Foto-Convention.md`.

---

## Fallback-Kaskade (wenn kein item-spezifisches Foto da)

```
Stufe 1: Item-Foto suchen
   /Content_Socialmedia/Fotos/{year}/PF_{year}_{itemId}_*.{jpg|png}
   gefunden? → ✓ verwenden

Stufe 2: Stock-Pool
   /Content_Socialmedia/Fotos/Instafotos Stock/
   zufällig eins wählen
   field_13 loggen: "Stock-Foto verwendet"

Stufe 3: Letzter Notnagel
   {REPO_RAW}/04_Canva-Vorlagen/html-templates/assets/judith-behandlung-01.png
   Teams-Card an Judith: "Bitte für PF-{year}-{itemId} ein Foto nachreichen"
   field_13 loggen: "Default-Foto verwendet"
```

## Neue Nodes in WF-02 (vor „Code: HTML bauen")

### Node 1: `Code: Foto-Pfad bauen`

Berechnet Suchparameter und gibt sie an die nachfolgende SP-Suche weiter.

```javascript
const f = $('Code: HWG-Filter').first().json;
const itemId = String($('Code: Validate + Route').first().json._sp_item_id).padStart(3, '0');

// Jahr aus Veröffentlichungsdatum (oder aktuelles Jahr als Fallback)
const datumRaw = f.sp_fields.Ver_x00f6_ffentlichungsdatum;
const year = datumRaw ? new Date(datumRaw).getFullYear() : new Date().getFullYear();

const sp_search_path = `/Content_Socialmedia/Fotos/${year}/`;
const filename_prefix = `PF_${year}_${itemId}_`;

return [{
  json: {
    itemId,
    year,
    sp_search_path,
    filename_prefix,
    // Original-Daten durchreichen
    ...$('Code: HWG-Filter').first().json,
  }
}];
```

### Node 2: `SP: Fotos im Ordner listen`

n8n-Node: **Microsoft SharePoint** → Operation `File: getMany`
- Site/Drive: dieselbe wie bestehende SP-Reads
- Path: `={{ $json.sp_search_path }}`
- Filter (optional): wird in nächstem Code-Node gefiltert

Output: Liste aller Dateien im Jahresordner.

### Node 3: `Code: Foto-Match`

Sucht in der Liste die passende Datei.

```javascript
const items = $input.all();
const cfg = $('Code: Foto-Pfad bauen').first().json;
const prefix = cfg.filename_prefix;

// Match-Versuch
const matches = items
  .map(i => i.json)
  .filter(f => (f.name || '').startsWith(prefix))
  .sort((a, b) => (a.name || '').localeCompare(b.name || ''));

if (matches.length > 0) {
  return [{
    json: {
      ...cfg,
      photo_found: true,
      photo_source: 'item',
      photo_sp_id: matches[0].id,
      photo_name: matches[0].name,
    }
  }];
}

// Fallback Stufe 2: Stock-Pool
// (wird im nächsten Sub-Branch behandelt — siehe Spec)
return [{
  json: {
    ...cfg,
    photo_found: false,
    photo_source: 'fallback',
    photo_name: null,
  }
}];
```

### Node 4: `SP: Foto herunterladen` (binary)

n8n-Node: **Microsoft SharePoint** → Operation `File: get`
- Item ID: `={{ $json.photo_sp_id }}`
- Property: `content` (binary)

Output: Binary unter `data`.

Conditional: Nur ausführen wenn `$json.photo_found === true`.

### Node 5: `Code: Foto → GitHub Body`

Analog zu IG/FB → GitHub Body aus v18.2.

```javascript
const binary = $input.first().binary?.data;
if (!binary) throw new Error('Kein Foto-Binary');

const cfg = $('Code: Foto-Match').first().json;
const repo_path = `04_Canva-Vorlagen/html-templates/assets/photos/${cfg.photo_name}`;
const message = `WF-02 v18.3: Foto-Upload für PF-${cfg.year}-${cfg.itemId}`;

return [{
  json: {
    repo_path,
    filename: cfg.photo_name,
    public_url: `https://raw.githubusercontent.com/twaese/Physio-Fuchs-Social-Media-Automation/main/${repo_path}`,
    github_body: { message, content: binary.data }
  }
}];
```

### Node 6: `HTTP: GitHub-Push Foto`

PUT auf `https://api.github.com/repos/twaese/Physio-Fuchs-Social-Media-Automation/contents/{{ $json.repo_path }}`
- Authorization: `Bearer {{ $env.GITHUB_TOKEN }}`
- Accept: `application/vnd.github+json`
- Body: `{{ JSON.stringify($json.github_body) }}`

⚠️ **Idempotent:** Falls Foto schon im Repo existiert, gibt GitHub `422 Unprocessable Entity` zurück.
→ Continue-on-fail in n8n-Settings aktivieren ODER vorher per GET prüfen ob File existiert.

Public-URL aus dem vorhandenen `$json.public_url` (egal ob Push lief oder schon da war).

## Updates am bestehenden `Code: HTML bauen (IG+FB)` Node

`vars`-Objekt um `BACKGROUND_PHOTO` erweitern:

```javascript
const fotoNode = $('HTTP: GitHub-Push Foto').first().json;  // public_url

const BACKGROUND_PHOTO_FALLBACK = `${REPO_RAW}/04_Canva-Vorlagen/html-templates/assets/judith-behandlung-01.png`;

const vars = {
  TITEL: f.titel,
  TEXT: f.text,
  CTA: f.cta,
  DATUM: datum,
  BACKGROUND_PHOTO: fotoNode?.public_url || BACKGROUND_PHOTO_FALLBACK,
};
```

`substitute(html, vars)` ersetzt `{{BACKGROUND_PHOTO}}` im Template (geändert in v13/FB v2).

## Edge Cases

| Fall | Verhalten |
|---|---|
| Foto > 10 MB | GitHub-API lehnt ab → Fallback aktiviert, Log: „Foto zu groß" |
| Foto mit Sonderzeichen im Namen (`Füße.jpg`) | Judith muss umbenennen — wird in `Foto-Convention.md` dokumentiert |
| Mehrere Fotos für gleiche ItemID | Alphabetisch erstes nehmen (vorhersehbar) |
| SP-Connector down | Workflow scheitert → n8n-Retry 3× → falls weiter rot: Teams-Alert |
| Foto in falschem Jahresordner | Item wird mit Fallback gepostet, Judith bekommt Hinweis |
